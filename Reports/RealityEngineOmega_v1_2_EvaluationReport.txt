FINDING V1-2-01 | SEVERITY: PHYSICS_BUG | PRIORITY: HIGH
SYSTEM: FQuantumGrid::UpdateSchrodingerEquation()
--------------------------------------------------------------------------------
TITLE: TDSE kinetic operator has three mathematical errors — propagator not unitary

DESCRIPTION:
  The TDSE is: iħ ∂Ψ/∂t = ĤΨ = [-ħ²/2m ∇² + V]Ψ
  The kinetic operator Ĥ_K acts on Ψ. It does not produce a scalar energy
  that is then divided by |Ψ|.

CURRENT_CODE:
  double d2R = (PsiReal[i+1] + PsiReal[i-1] - 2.0 * PsiReal[i]);
  double PsiMag = FMath::Max(FMath::Sqrt(PsiReal[i]*PsiReal[i] + PsiImag[i]*PsiImag[i]), 1e-9);
  double KineticE = -(Hbar*Hbar / (2.0*Mass)) * (d2R / PsiMag);
  double Phase = -(KineticE + V) * DeltaTime / Hbar;

ERROR_1 — Division by |Ψ| (no physical basis):
  d2R / PsiMag has no meaning in quantum mechanics. The kinetic operator
  maps a wave function Ψ to another wave function KΨ. Dividing by |Ψ|
  is not part of that operation. This is the primary error.

ERROR_2 — Imaginary Laplacian ignored:
  Only d2R (∂²PsiReal/∂x²) is computed. The full discrete Laplacian
  requires d2I (∂²PsiImag/∂x²) as well. Ignoring it introduces an
  asymmetry between real and imaginary evolution that breaks unitarity
  — the wave function norm |Ψ|² is not conserved over time.

ERROR_3 — Missing grid spacing dx²:
  The discrete 1D Laplacian is: ∇²f ≈ (f[i+1] + f[i-1] - 2f[i]) / dx²
  The code omits the dx² denominator. Without it, the kinetic energy
  contribution is off by a factor of 1/dx². FQuantumGrid has GridSize
  but no defined spatial extent, so dx is implicitly 1 — this should
  be made explicit.

CONSEQUENCE:
  The propagator no longer conserves |Ψ|² (not unitary). Wave function
  norms drift over time. The v1.1 "1.0 + V" phase rotation was cruder
  but self-consistent as a diagonal rotation (unitary by construction).
  V1.2 is less physically accurate than v1.1 despite appearing more rigorous.

CORRECT_IMPLEMENTATION (Euler method for TDSE):
  The TDSE split into real and imaginary parts gives:
    dR/dt = +(1/ħ) * [Ĥ_K applied to PsiImag + V * PsiImag]
    dI/dt = -(1/ħ) * [Ĥ_K applied to PsiReal + V * PsiReal]

  CORRECT_CODE:
    // Add FQuantumGrid member: double GridSpacing = 1.0; (default 1 unit)
    double dx2 = GridSpacing * GridSpacing;
    double d2R = (PsiReal[i+1] + PsiReal[i-1] - 2.0*PsiReal[i]) / dx2;
    double d2I = (PsiImag[i+1] + PsiImag[i-1] - 2.0*PsiImag[i]) / dx2;
    double KR  = -(Hbar*Hbar / (2.0*Mass)) * d2R;  // K applied to real part
    double KI  = -(Hbar*Hbar / (2.0*Mass)) * d2I;  // K applied to imag part
    double dt  = static_cast<double>(DeltaTime);
    double newR = PsiReal[i] + (dt / Hbar) * (KI + V * PsiImag[i]);
    double newI = PsiImag[i] - (dt / Hbar) * (KR + V * PsiReal[i]);
    PsiReal[i] = newR;
    PsiImag[i] = newI;

NOTE: Euler method is first-order and will accumulate phase error over long
simulations. For production, Runge-Kutta 4th order or the Crank-Nicolson
method provides better norm conservation. For a game engine, Euler is
acceptable if DeltaTime is small.

STATUS: OPEN — requires fix before V1.3

FINDING V1-2-03 | SEVERITY: ENGINEERING_BUG | PRIORITY: MEDIUM
SYSTEM: ProcessLocalCurvature() — field decay loop
--------------------------------------------------------------------------------
TITLE: Curvature field decay is frame-rate dependent

DESCRIPTION:
  The decay factor 0.95 is applied once per tick regardless of DeltaTime.
  Physical dissipation is a continuous process with a characteristic decay
  rate — it should be parameterized by time elapsed, not tick count.

QUANTIFIED_IMPACT:
  At 60fps (dt ≈ 0.0167s): field halves in ~13.5 ticks = ~0.22 seconds
  At 30fps (dt ≈ 0.0333s): field halves in ~13.5 ticks = ~0.45 seconds
  At 15fps (dt ≈ 0.0667s): field halves in ~13.5 ticks = ~0.90 seconds
  The curvature field persists 4x longer at 15fps than 60fps.
  Render output (MaxLocalCurvature in FSimulationState) is inconsistent
  across hardware targets.

CURRENT_CODE:
  for (double& C : LocalCurvatureField) { C *= 0.95; }

CORRECT_CODE:
  // Choose decay rate so field halves in ~0.23 seconds at any framerate
  // Half-life = ln(2) / DecayRate → DecayRate = ln(2) / 0.23 ≈ 3.0
  constexpr double DecayRate = 3.0;
  double DecayFactor = FMath::Exp(-DecayRate * static_cast<double>(DeltaTime));
  for (double& C : LocalCurvatureField) { C *= DecayFactor; }

STATUS: OPEN — fix recommended before any framerate-variable deployment

FINDING V1-2-04 | SEVERITY: MINOR_BUG | PRIORITY: LOW
SYSTEM: ProcessLocalCurvature() — CurrentMaxCurvature tracking
--------------------------------------------------------------------------------
TITLE: CurrentMaxCurvature reports 0 when no actors present — decaying field ignored

DESCRIPTION:
  CurrentMaxCurvature is reset to 0.0 at the top of ProcessLocalCurvature().
  It is only updated when an actor writes to a voxel in the injection loop.
  When ActiveActors is empty, the function exits with CurrentMaxCurvature = 0
  even though LocalCurvatureField may contain significant non-zero values
  from previous frames (reduced by 0.95 decay but not zero).

  GetSimulationState() then reports MaxLocalCurvature = 0, which incorrectly
  tells external renderers the curvature field is empty when it is not.

CURRENT_BEHAVIOR:
  Frame N:   Actor present → voxel written → MaxLocalCurvature = X
  Frame N+1: Actor removed → no voxel written → MaxLocalCurvature = 0
             (field still contains ~0.95X from decay, but unreported)

CORRECT_FIX:
  After the injection loop, scan the full field or accumulate during decay:

  Option A — scan after injection (simple):
    for (const double& C : LocalCurvatureField)
        CurrentMaxCurvature = FMath::Max(CurrentMaxCurvature, C);

  Option B — track during decay (no extra pass, recommended):
    for (double& C : LocalCurvatureField)
    {
        C *= DecayFactor; // (after applying V1-2-03 fix)
        CurrentMaxCurvature = FMath::Max(CurrentMaxCurvature, C);
    }

STATUS: OPEN — minor fix, no crash risk, affects render interface accuracy

FINDING V1-2-02 | SEVERITY: ADVISORY | PRIORITY: LOW
SYSTEM: CycleEnhancedRand()
--------------------------------------------------------------------------------
TITLE: reinterpret_cast type punning — undefined behaviour under C++ strict aliasing

DESCRIPTION:
  Reading a float's bit pattern via a uint32* pointer is undefined behaviour
  under the C++ standard's strict aliasing rule (N4860 §6.7.1). Both MSVC
  and UE5-configured Clang use -fno-strict-aliasing and handle this correctly
  in practice, but the code will generate warnings with UBSan or static
  analyzers and is technically non-conformant.

CURRENT_CODE:
  uint32 Mixed = static_cast<uint32>(Cycles) ^ *reinterpret_cast<uint32*>(&FRandVal);

CORRECT_CODE (standard-conformant, identical output):
  uint32 FRandBits;
  FMemory::Memcpy(&FRandBits, &FRandVal, sizeof(uint32));
  uint32 Mixed = static_cast<uint32>(Cycles) ^ FRandBits;

NOTE: The XOR logic and resulting uniform distribution are correct regardless.
This is a code quality / conformance fix only — no functional difference.

STATUS: ADVISORY — fix recommended for UBSan compatibility and code audits

    range are clamped to the boundary voxel — no out-of-bounds warning issued.
