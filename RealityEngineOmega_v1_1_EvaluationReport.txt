FINDING V1-01 | SEVERITY: PHYSICS_BUG | PRIORITY: HIGH
SYSTEM: HardwareTRNG() → GaussianNoise() → ProcessKPZGrowth()
--------------------------------------------------------------------------------
TITLE: HardwareTRNG() produces non-uniform distribution — biases KPZ noise spectrum

DESCRIPTION:
  HardwareTRNG() adds two uniform variables then applies FMod:
    FMath::FRand()        → uniform U[0.0, 1.0]
    CycleNoise            → uniform U[0.0, 0.9999]  (Cycles % 10000 / 10000.0f)
    Sum                   → U[0.0, 1.9999]
    FMath::Fmod(Sum, 1.0) → wraps values >1 back to [0, 1)

  The sum of two independent uniform variables produces a TRIANGULAR distribution
  peaked at 0.5, not a uniform distribution. FMod does not correct this — it
  folds the triangle, doubling the probability density near 0 and 1 and leaving
  0.5 with twice the density of the extremes.

DOWNSTREAM_IMPACT:
  The Box-Muller transform assumes inputs U1, U2 are independent uniform [0,1].
  With triangularly-distributed inputs, the output Z is no longer Gaussian.
  KPZ theory requires Gaussian white noise:
    <η(x,t)η(x',t')> = 2D δ(x-x') δ(t-t')
  Biased noise produces a non-flat power spectrum, shifting KPZ surface growth
  exponents away from their correct universality class values:
    Correct KPZ (1+1)D: α=1/2, β=1/3, z=3/2
    Biased result:      exponents drift, surface roughness incorrect

SECONDARY_ISSUE:
  The function is named HardwareTRNG but is not hardware TRNG.
  CPU cycle counters (FPlatformTime::Cycles64) are deterministic hardware
  registers, not physically random sources. True TRNG requires thermal noise
  or shot noise (e.g., GPU voltage jitter via NVML/ADL).
  Current implementation is more accurately: cycle-seeded enhanced PRNG.
  The naming will mislead future developers and documentation consumers.

CURRENT_CODE:
  uint64 Cycles = FPlatformTime::Cycles64();
  float CycleNoise = static_cast<float>(Cycles % 10000) / 10000.0f;
  return FMath::Fmod(FMath::FRand() + CycleNoise, 1.0f);

FIX_OPTION_A (blend without folding):
  float U = FMath::FRand() * 0.5f + (CycleNoise * 0.5f);
  return FMath::Clamp(U, 1e-7f, 1.0f - 1e-7f);

FIX_OPTION_B (bitwise XOR mixing — preserves full entropy):
  float FRandVal = FMath::FRand();
  uint32 Mixed = static_cast<uint32>(Cycles) ^ *reinterpret_cast<uint32*>(&FRandVal);
  return static_cast<float>(Mixed) / static_cast<float>(MAX_uint32);

RENAME_RECOMMENDATION:
  HardwareTRNG() → CycleEnhancedRand() until NVML integration is live.
  Rename to HardwareTRNG() when GPU thermal/voltage telemetry is the source.

STATUS: OPEN — fix required before NVML production swap

--------------------------------------------------------------------------------
TITLE: Unit Hamiltonian kinetic placeholder — "1.0" approximation persists

DESCRIPTION:
  A-01 (free-particle Hamiltonian) is resolved: V(x,t) is now incorporated.
  The remaining limitation is that the kinetic energy term is hardcoded as 1.0:
    Phase = -(1.0 + V) * dt / Hbar

  The dimensionally correct kinetic operator for a particle of mass m is:
    T = -ħ²∇²Ψ / 2m   (varies per grid cell based on wave function curvature)

  The "1.0" substitutes a fixed 1 Joule for the per-cell kinetic energy.
  This is dimensionally consistent (1.0 [J] + V [J]) / Hbar [J·s] = [s⁻¹)
  but physically approximate — every grid cell is treated as having exactly
  the same kinetic energy regardless of wave function shape.

IMPACT:
  Meaningful improvement over the V3 free-particle propagator (V=0 everywhere).
  The remaining inaccuracy affects the relative phase evolution speed between
  grid cells, not the overall correctness of collapse/decoherence logic.
  For a game-engine physics simulator this is an acceptable scope limitation.

FUTURE_FIX (v1.2 recommendation):
  Compute Laplacian of PsiReal/PsiImag across the grid, multiply by
  -CONST_HBAR² / (2 * Mass), add to V[i] for the full Hamiltonian:
    double KineticTerm = -(CONST_HBAR * CONST_HBAR / (2.0 * Mass)) * Laplacian_Psi[i];
    double Phase = -(KineticTerm + V) * dt / Hbar;

STATUS: ADVISORY — no action required for current scope
